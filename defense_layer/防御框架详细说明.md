# 自适应防御框架详细说明

## 1. 框架定位

这个防御框架是一个面向**多轮对话场景**的轻量级自适应防御原型，核心目标不是直接复刻某一篇论文的完整训练系统，而是把开题报告中的三类核心思想工程化落地：

1. 上下文状态建模
2. 双向意图推断
3. 分级防御策略执行

对应代码入口：

1. `adaptive_defense_framework/context_state.py`
2. `adaptive_defense_framework/intent_inference.py`
3. `adaptive_defense_framework/defense_policy.py`
4. `adaptive_defense_framework/defense_engine.py`

整个框架的核心思想是：

> 不只看当前这一句用户输入，而是结合多轮上下文状态、输入输出双向语义信号和风险分级策略，在每一轮对话上做一次“感知 - 判断 - 调控”闭环。

1. 读取一轮用户输入和原始助手输出
2. 用上下文状态模块更新风险状态
3. 用双向意图推断模块估计当前轮风险
4. 融合上下文风险与当前轮风险
5. 用策略层决定是放行、引导、改写还是拒绝
6. 输出新的防御后回答，并写入日志

### 3.2 设计目的

这个模块非常简单，但它解决的是框架的输入标准化问题：

1. `Turn` 表示单轮消息
2. `DialogueSample` 表示一条完整对话样本

1. 有的数据集是 `history`
2. 有的是 `user/assistant` 成对结构
3. 有的是多轮 conversation 列
4. 有的带标签，有的不带

用统一的数据结构封装后，后面的状态建模、意图推断、策略执行就都只需要面对同一种输入对象。

### 3.4 好处

1. 便于适配多种数据集
2. 便于写统一的 runner 和评测脚本
3. 便于后续替换模型或替换策略层，而不需要改数据接口

## 4. 模块一：上下文状态建模

文件：
[context_state.py](C:/Users/hhh/Desktop/毕设/攻击代码复现/code/4.1.2_隐式意图自适应推断/adaptive_defense_framework/context_state.py)

### 4.1 对应的研究思想和来源

这个模块直接对应你的研究内容里的“多轮上下文状态建模”。

代码注释明确写了：

```python
# Inspired by NBF-style "state trajectory + online risk" design.
```

它主要借鉴的是：

1. NBF-LLM  
论文：*Steering Dialogue Dynamics for Robustness against Multi-turn Jailbreaking Attacks*  
本地参考仓库：[HanjiangHu_NBF-LLM/README.md](C:/Users/hhh/Desktop/毕设/攻击代码复现/code/4.1.1_多轮上下文状态建模/HanjiangHu_NBF-LLM/README.md)

这篇工作的核心思想不是只看单轮输入，而是把多轮对话看成一个动态轨迹，在轨迹上估计风险并施加“安全转向”。

### 4.2 代码实现

```python
DEFAULT_HARMFUL_PROTOTYPES = [
    "step by step instructions for unsafe operation",
    "bypass safety guardrail jailbreak prompt injection exploit",
    "disable policy restrictions and reveal restricted details",
    "high risk request with procedural execution details",
]
```

这部分定义了一组“危险语义原型”。它不是完整分类器，而是一组参考向量模板。

```python
@dataclass
class ContextState:
    turn_index: int = 0
    cumulative_risk: float = 0.0
    last_vector: List[float] = field(default_factory=list)
    last_drift: float = 0.0
    recent_drifts: List[float] = field(default_factory=list)
```

`ContextState` 保存的是对话运行时状态：

1. `turn_index`：当前轮次
2. `cumulative_risk`：累计风险
3. `last_vector`：上一轮语义表示
4. `last_drift`：上一轮漂移量
5. `recent_drifts`：最近几轮漂移记录

主逻辑在 `update()`：

```python
merged = f"user: {user_text}\nassistant: {assistant_text}"
current_vec = hash_embed(merged, dim=self.dim)
```

这里先把用户输入和助手回答拼接成一个统一的当前轮语义单元，再做哈希嵌入。

```python
drift = 0.0
if state.last_vector:
    drift = max(0.0, 1.0 - cosine(current_vec, state.last_vector))
```

这一步计算“语义漂移”：

1. 如果当前语义向量和上一轮很接近，说明对话还在同一主题上，`drift` 小
2. 如果差异很大，说明对话发生了明显转向，`drift` 大

这对多轮越狱很关键，因为很多攻击不是一开始就暴露恶意，而是逐轮迁移意图。

```python
prototype_sims = [cosine(current_vec, p) for p in self.prototype_vectors]
danger_similarity = max(prototype_sims) if prototype_sims else 0.0
```

这里计算“当前轮和危险原型的最大相似度”，也就是：

1. 当前语义离危险语义模板有多近
2. 即使显式关键词不多，只要整体语义方向接近危险区域，也能得到一定风险信号

```python
state.recent_drifts.append(drift)
state.recent_drifts = state.recent_drifts[-6:]
drift_trend = mean(state.recent_drifts)
```

这里不是只看一轮，而是保留最近 6 轮的漂移均值，用于构造“漂移趋势”。

### 4.3 为什么这样设计

传统单轮防御的问题是：

1. 只看当前句子
2. 容易漏掉逐步诱导
3. 无法表达“当前轮其实是前几轮铺垫出来的”

所以这个模块引入三类状态：

1. 局部状态：`drift`
2. 趋势状态：`drift_trend`
3. 危险邻近度：`danger_similarity`

它们共同构成“对话现在处在什么风险区域”的判断基础。

### 4.4 这么设计的好处

1. 能处理多轮上下文，而不是只做单轮文本分类
2. 能捕捉“逐轮偏移”的攻击模式
3. 计算开销低，不依赖大模型额外推理
4. 容易在线更新，适合实时对话防御

### 4.5 当前实现的工程取舍

这里使用的是 `hash_embed()` 生成的稀疏哈希向量，而不是论文里更强的神经表示学习。

相关代码在：
[text_features.py](C:/Users/hhh/Desktop/毕设/攻击代码复现/code/4.1.2_隐式意图自适应推断/adaptive_defense_framework/text_features.py)

```python
def hash_embed(text: str, dim: int = 1024) -> List[float]:
    vec = [0.0] * dim
    tokens = tokenize(text)
    ...
    idx = int(hashlib.md5(tok.encode("utf-8")).hexdigest(), 16) % dim
    vec[idx] += float(cnt)
```

这意味着：

1. 它是轻量实现
2. 它适合工程验证
3. 但它不是 NBF-LLM 的完整神经动态系统

所以更准确的表述应该是：

> 这个模块是“受 NBF-LLM 启发的简化多轮状态建模实现”，而不是原论文方法的完全复现。

### 5.2 正向意图推断

```python
FORWARD_KEYWORDS: Dict[str, float] = {
    "step by step": 0.24,
    "detailed instructions": 0.20,
    "how to": 0.18,
    "bypass": 0.20,
    "exploit": 0.20,
    "jailbreak": 0.20,
    "ignore safety": 0.20,
    ...
}
```

这部分定义了用户输入侧的风险词表。它体现的思想是：

1. 用户请求如果包含过程化、规避式、解限制式表达
2. 就要对输入意图提高警惕

核心函数：

```python
def forward_intent(self, user_text: str) -> float:
    base = normalize_text(user_text).lower()
    if not base:
        return 0.0
    k = self._smooth_keyword_score(base, FORWARD_KEYWORDS, n=5)
    cmd = 0.16 if COMMAND_PATTERN.search(base) else 0.0
    progressive = 0.08 if "continue" in base or "next step" in base else 0.0
    return min(1.0, k + cmd + progressive)
```

这里做了三件事：

1. `k`：关键词匹配分数
2. `cmd`：命令式语气加分
3. `progressive`：多轮推进信号加分

#### 为什么要单独加 `progressive`

多轮越狱常见特点不是直接问，而是：

1. 继续
2. 下一步
3. 接着讲
4. 更具体一点

这类词本身未必危险，但在多轮上下文里常常意味着攻击正在推进。因此这里把它单独建模。

### 5.3 反向意图推断

```python
BACKWARD_KEYWORDS: Dict[str, float] = {
    "first,": 0.08,
    "second,": 0.08,
    "third,": 0.08,
    "steps": 0.12,
    "materials": 0.08,
    "exactly": 0.08,
    ...
}
```

这里不是看用户，而是看模型回答本身是否已经朝着“可执行步骤化输出”发展。

核心函数：

```python
def backward_intent(self, assistant_text: str) -> float:
    base = normalize_text(assistant_text).lower()
    if not base:
        return 0.0
    k = keyword_score(base, BACKWARD_KEYWORDS)
    numbered_steps = 0.12 if re.search(r"(^|\n)\s*\d+[\.\)]", assistant_text) else 0.0
    long_procedural = 0.12 if len(base.split()) > 120 and ("step" in base or "instructions" in base) else 0.0
    return min(1.0, k + numbered_steps + long_procedural)
```

这一步的判断逻辑是：

1. 是否出现步骤词
2. 是否出现编号步骤
3. 是否出现较长程序化说明

也就是说，它不再问“用户是不是危险”，而是问：

> 当前模型回答是不是已经在走向可执行的危险输出

### 5.4 SmoothLLM 启发：平滑估计

```python
def _smooth_keyword_score(self, text: str, keyword_weights: Dict[str, float], n: int = 5) -> float:
    base = normalize_text(text)
    ...
    for _ in range(n):
        perturbed = chars[:]
        if len(perturbed) > 8:
            idx = self.random.randint(0, len(perturbed) - 1)
            perturbed[idx] = " "
        ptxt = "".join(perturbed)
        scores.append(keyword_score(ptxt, keyword_weights))
    return sum(scores) / len(scores)
```

这一步借鉴 SmoothLLM 的关键思想：

1. 对输入做轻微扰动
2. 不只看单次判断
3. 看扰动后风险信号是否稳定存在

虽然这里不是 SmoothLLM 原论文里的完整随机扰动投票机制，但核心思想是一致的：

> 一个真正稳定的恶意信号，不应该因为一个字符被替换成空格就完全消失。

### 5.5 风险融合

```python
def fuse(self, forward_score: float, backward_score: float, drift_signal: float, cumulative_risk: float) -> float:
    fused = (
        0.36 * forward_score
        + 0.30 * backward_score
        + 0.20 * drift_signal
        + 0.14 * cumulative_risk
    )
    return min(1.0, max(0.0, fused))
```

这里把风险拆成四部分：

1. 用户输入侧意图
2. 模型回答侧意图
3. 当前上下文漂移/危险相似度
4. 历史累计风险

### 5.6 为什么这样设计

单独看任一信号都不够：

1. 只看输入，可能被伪装
2. 只看输出，可能为时已晚
3. 只看上下文，可能漏掉当前轮强恶意
4. 只看当前轮，可能漏掉多轮推进

所以模块二的设计思想是：

> 用双向意图 + 上下文动态 + 历史风险，构成一个多信号融合判断器。

### 5.7 好处

1. 比单轮关键词过滤更鲁棒
2. 能对“看起来温和、实际在推进”的输入更敏感
3. 能对“模型输出已经变危险”的情况做补救
4. 易于扩展，后续可以替换成分类器或小模型评分器

### 5.8 当前实现的局限

这个模块当前仍然是**规则驱动**而不是学习式双向推断器，因此：

1. 风险词覆盖不全
2. 语义泛化有限
3. 对显式危险类别的覆盖还不够强

所以它更适合做“概念落地版”和“答辩演示版”，而不是最终工业版。

## 6. 模块三：分级防御策略

文件：
[defense_policy.py](C:/Users/hhh/Desktop/毕设/攻击代码复现/code/4.1.2_隐式意图自适应推断/adaptive_defense_framework/defense_policy.py)

### 6.1 对应的研究思想和来源

代码注释写明：

```python
# Inspired by X-Boundary / policy-tiered control:
# normal -> guide -> rewrite -> partial_refuse -> refuse
```

它对应的是你的研究内容里的“防御策略层设计”。

本地最直接的参考论文是：

1. X-Boundary  
论文：*X-Boundary: Establishing Exact Safety Boundary to Shield LLMs from Multi-Turn Jailbreaks without Compromising Usability*  
本地参考仓库：[AI45Lab_X-Boundary/README.md](C:/Users/hhh/Desktop/毕设/攻击代码复现/code/4.1.3_防御策略层设计/AI45Lab_X-Boundary/README.md)

这个方向强调的核心不是“一刀切拒绝”，而是：

1. 尽可能保留正常可用性
2. 只在越过危险边界时提高干预强度

### 6.2 分级阈值

```python
self.level_thresholds = {
    "low": 0.25,
    "medium": 0.45,
    "high": 0.70,
}
```

这里把风险分成四段：

1. `low`
2. `medium`
3. `high`
4. `critical`

这意味着策略层不是二分类，而是按风险等级逐步加码。

### 6.3 决策入口

```python
def decide(self, risk_score: float, user_text: str, assistant_text: str) -> DefenseDecision:
    level = self.level_of(risk_score)
    if level == "low":
        return DefenseDecision("allow", level, assistant_text)
    if level == "medium":
        return DefenseDecision("guide", level, self._guidance(assistant_text))
    if level == "high":
        rewritten = self._sanitize_response(assistant_text)
        return DefenseDecision("rewrite", level, rewritten)
    if self._contains_sensitive(user_text):
        return DefenseDecision("partial_refuse", level, self._partial_refusal(user_text))
    return DefenseDecision("refuse", level, self._refusal())
```

这段代码就是整个框架的“调控器”。

它表达的是非常清晰的策略逻辑：

1. 低风险：放行
2. 中风险：引导
3. 高风险：改写
4. 极高风险：部分拒绝或完全拒绝

### 6.4 引导策略 `guide`

```python
def _guidance(self, text: str) -> str:
    base = normalize_text(text)
    if not base:
        base = "..."
    return (
        "为确保内容安全与合规，以下回答仅保留原理性、教育性信息：\n"
        f"{base}"
    )
```

这个策略并不直接截断，而是把原回答重新包上一层“仅保留原理性信息”的语义框架。

#### 为什么这么设计

因为很多场景不是纯恶意，而是模糊边界：

1. 研究讨论
2. 教育背景
3. 风险说明
4. 合规解释

在这种情况下直接拒绝会损失可用性，所以先做引导。

### 6.5 改写策略 `rewrite`

```python
def _sanitize_response(self, text: str) -> str:
    content = normalize_text(text)
    ...
    content = re.sub(r"(^|\n)\s*\d+[\.\)]\s*", "\n- ", content)
    content = re.sub(
        r"\b(exploit|payload|bypass|jailbreak|illegal|unsafe)\b",
        "[敏感术语已脱敏]",
        content,
        flags=re.IGNORECASE,
    )
    content = re.sub(r"\b\d+(\.\d+)?\s*(ml|g|kg|%)\b", "[参数已脱敏]", content, flags=re.IGNORECASE)
```

这里做的是“内容脱敏改写”：

1. 去掉明显步骤编号
2. 抹掉敏感术语
3. 抹掉具体参数

#### 为什么这样设计

因为有一类回答并不是必须完全拒绝，只要把：

1. 可执行细节去掉
2. 具体参数去掉
3. 操作步骤去掉

就可以保留较高层次信息。

### 6.6 部分拒绝和完全拒绝

```python
def _partial_refusal(self, user_text: str) -> str:
    return (
        "你的请求中包含高风险部分，我无法提供相关可执行细节。"
        "如果你愿意，我可以继续提供该主题的安全背景、法规与防护建议。"
    )
```

```python
def _refusal(self) -> str:
    return "抱歉，我不能协助这类高风险请求。可以换一个安全、合规的话题。"
```

这里体现的是两层拒绝：

1. `partial_refuse`：拒绝危险部分，但保留安全替代方向
2. `refuse`：完全拒绝

### 6.7 为什么这样设计

如果所有风险一律拒绝，会出现严重过拒答。  
如果所有风险只做引导，又防不住真实攻击。

因此最合理的工程路径是：

1. 风险低时保留可用性
2. 风险中等时做柔性干预
3. 风险高时做内容改写
4. 风险极高时拒绝

### 6.8 这么设计的好处

1. 符合真实产品安全系统的分级控制思路
2. 平衡了安全性和可用性
3. 容易做实验 ablation
4. 易于和多轮风险评分模块解耦

### 7.2 初始化

```python
        self.state_model = ContextStateModel(dim=1024)
        self.intent = DualIntentInferencer()
        self.policy = AdaptiveDefensePolicy()
```

这里说明整个系统是由三个独立子模块组成：

1. 状态建模器
2. 意图推断器
3. 策略控制器

### 7.3 主循环

```python
while i < len(sample.turns):
    turn = sample.turns[i]
    if turn.role != "user":
        defended_turns.append(turn)
        i += 1
        continue
```

系统逐轮扫描对话，只在遇到 `user` 时触发一次防御判断。

```python
user_text = turn.content
original_assistant = ""
next_is_assistant = i + 1 < len(sample.turns) and sample.turns[i + 1].role == "assistant"
if next_is_assistant:
    original_assistant = sample.turns[i + 1].content
```

这里把一轮定义为：

1. 当前用户输入
2. 紧随其后的原始助手输出

### 7.4 状态更新

```python
context_info = self.state_model.update(state, user_text=user_text, assistant_text=original_assistant)
```

模块一先工作，输出：

1. `drift`
2. `drift_trend`
3. `danger_similarity`
4. `turn_index`

### 7.5 双向意图估计

```python
forward = self.intent.forward_intent(user_text)
backward = self.intent.backward_intent(original_assistant)
```

模块二随后工作，分别给出：

1. 输入风险
2. 输出风险

### 7.6 风险融合与累计更新

```python
fused = self.intent.fuse(
    forward_score=forward,
    backward_score=backward,
    drift_signal=max(context_info["drift"], context_info["danger_similarity"]),
    cumulative_risk=state.cumulative_risk,
)
state.cumulative_risk = min(1.0, 0.75 * state.cumulative_risk + 0.25 * fused)
```

这一段非常关键。

它说明整个系统不是“每轮独立判断”，而是：

1. 当前轮会参考历史累计风险
2. 历史累计风险又会吸收当前轮的新风险

这就形成了一个在线闭环状态更新器。

### 7.7 策略决策

```python
decision = self.policy.decide(
    risk_score=fused,
    user_text=user_text,
    assistant_text=original_assistant,
)
```

然后模块三根据融合风险决定最终动作。

### 7.8 写回防御后的对话

```python
defended_turns.append(Turn(role="user", content=user_text))
defended_turns.append(Turn(role="assistant", content=decision.message))
```

这里把原始对话改造成“防御后的对话”，为后续评测和导出提供统一结果。

### 7.9 日志记录

```python
turn_logs.append(
    {
        "sample_id": sample.sample_id,
        "turn_id": len(turn_logs) + 1,
        "user_text": user_text,
        "original_assistant": original_assistant,
        "defended_assistant": decision.message,
        "forward_intent": round(forward, 4),
        "backward_intent": round(backward, 4),
        "drift": round(context_info["drift"], 4),
        "danger_similarity": round(context_info["danger_similarity"], 4),
        "fused_risk": round(fused, 4),
        "cumulative_risk": round(state.cumulative_risk, 4),
        "action": decision.action,
        "risk_level": decision.risk_level,
    }
)
```

这一步是整个框架非常重要的工程价值所在。

因为它把每轮判断的全部中间变量都保存下来，你后续可以非常方便地做：

1. 错误分析
2. 画图
3. 论文表格
4. 策略对比实验

### 7.10 预测输出

```python
prediction = 1 if any(x["fused_risk"] >= 0.45 for x in turn_logs) else 0
```

这表示如果任意一轮达到 `medium` 阈值以上，就把整条对话视为有风险预测。

最终返回：

1. 原样本
2. 风险预测
3. 每轮日志
4. 防御后的完整对话
5. 最终累计风险

## 9. 为什么要这样串联

### 9.1 先感知，再判断，再调控

这个顺序非常合理，因为安全系统本质上就应该是：

1. 先知道发生了什么
2. 再判断风险有多大
3. 最后决定怎么处理

如果没有这个顺序，就会出现：

1. 没看清上下文就直接拒绝
2. 没看输出走向就误判
3. 没有风险等级就只能二选一

### 9.2 每一层职责单独清晰

这样设计的最大工程优势是**可替换性**。

例如：

1. 你以后想把 `hash_embed()` 换成 BERT embedding，只改模块一
2. 你想把关键词打分换成小模型分类器，只改模块二
3. 你想把规则策略换成 learned policy，只改模块三

整个系统不需要推倒重来。

### 9.3 既能做研究，也能做工程

这个框架的优势是两头兼顾：

1. 对研究：每个模块都能做消融实验
2. 对工程：每个模块都能独立调参和替换

### 9.4 适合论文叙述

从论文写作角度，这种串联方式也很好写：

1. 4.1.1：上下文状态建模
2. 4.1.2：双向意图推断
3. 4.1.3：策略层执行
4. 4.1.4：端到端防御流程与实验

结构非常自然。

## 11. 必须诚实说明的局限

虽然这个框架结构上是合理的，但目前实现仍然是一个**研究原型版**，不是最终强防御系统。

主要局限有：

1. 状态表示还是轻量哈希特征，不是学习式动力学模型
2. 意图推断还是规则驱动，不是训练出来的双向判别器
3. 风险原型和关键词覆盖面还有限
4. 策略层目前的 `guide` 会保留较多原始内容
5. 对显式危险请求的强拒答能力还不够

也正因为如此，它特别适合答辩中的表述方式应该是：

> 本系统实现了一个受 NBF-LLM、BIID、SmoothLLM 和 X-Boundary 启发的多轮自适应防御框架原型，重点在于完成从理论模块到工程系统的落地，并验证其端到端运行闭环。

## 12. 对答辩时的推荐讲法

最推荐的讲法是：

1. 先讲为什么单轮防御不够，要看多轮上下文
2. 再讲为什么只看输入也不够，要做双向意图推断
3. 再讲为什么不能全拒绝，要做分级策略
4. 最后讲为什么这三个模块串起来，能形成自适应闭环

一句话总括可以写成：

> 本文实现的防御框架以多轮状态感知为基础，以双向意图推断为判断核心，以分级策略控制为输出机制，构成了一个兼顾安全性、可解释性和可扩展性的多轮对话防御系统。
